# 制度依据双向复合复核工作流

版本：v2.0  
文档关系：本文件说明制度依据专项复核的具体流程；通用文本处理规则见 `文本处理规则与方法说明.md`，项目材料和结论见 `PROJECT_CONTEXT.md`，当前进度和续作入口见 `HANDOFF.md`。

## 一、工作流目标

本工作流用于同时解决两个问题：

1. 从制度出发，保证每项具有管理意义的制度义务都被转化为管理动作；
2. 从管理动作和执行计划反查，保证动作成体系、有依据，并在分类、合并过程中不丢失原条款。

工作流采用两条相互校验的链路：

- 正向链路：制度源文件 → 结构化条款 → 最小可执行义务 → 标准管理动作 → 执行计划任务；
- 反向链路：执行计划任务 → 标准管理动作 → 最小可执行义务 → 结构化条款 → 制度源文件。

两条链路通过“义务ID、条款ID、动作ID、任务序号”连接。任何分类或合并只能改变展示和归类，不得删除这些追溯关系。

本工作流默认不自动修改原执行计划。只有双向缺口关闭并经人工确认允许回写后，才能进入修改环节。

## 二、输入和输出

### （一）输入

- 全部外部监管文件及效力状态；
- 全部行内制度、实施细则、应急预案、附件和表单；
- 最新版外包商管理全流程执行计划；
- 采购文件、合同、交付物、验收和付款条件；
- 项目事实，包括外包类型、重要性、驻场、数据处理、AI使用、分包、关联和跨境情况。

### （二）输出

- 制度源文件清单；
- 制度条款结构化库；
- 原子义务清单；
- 标准管理动作库；
- 条款到动作正向矩阵；
- 动作到制度反向矩阵；
- 执行计划到标准动作映射；
- 双向覆盖缺口清单；
- 原制度依据问题清单；
- 人工确认与回写清单；
- 可重复运行的JSON数据和规则脚本。

## 三、四层数据模型

### 第一层：制度和条款

每份源文件记录来源ID、名称、文号、发布机构、层级、效力、版本、日期、路径、格式和提取状态。每条制度条文记录条款ID、原文及结构化状态。

已废止文件只保留版本沿革；征求意见稿只作前瞻参考；采购合同只能作为项目直接依据，不能冒充外部监管依据。

### 第二层：最小可执行义务

一条条文可能包含多个义务，必须拆到可以单独判断责任、对象、动作、时点和证据的粒度。每项义务生成稳定的义务ID，并逐项回答：

1. 谁做；
2. 对什么对象做；
3. 必须做什么；
4. 在什么时点或频率做；
5. 必须或建议留下什么证据；
6. 本项目直接适用、条件适用、机构层面适用还是不形成项目任务；
7. 能否直接推出管理动作，还是只能作为支持依据。

不得因同一原文在不同章节重复出现而静默删除条款ID。可以标记重复，但必须保留来源关系。

### 第三层：标准管理动作

将语义相近的原子义务归入标准动作，例如风险评估、尽职调查、合同审核、SLA管理、数据委托处理、AI场景识别、应急处置等。

每个标准动作至少记录：动作ID、管理类别、动作名称、动作类型、默认适用性、任务属性、动作说明、全部义务ID、全部条款依据、直接或条件依据、对应任务序号。

合并动作时执行“并集原则”：合并后的义务ID集合必须等于被合并记录义务ID的并集；不得只保留代表性条款。

### 第四层：执行计划任务

执行计划任务保留原序号，并映射一个或多个标准动作。映射结果同时利用任务文本和现有制度依据，不因其中一条链路失败而丢弃另一条链路。

## 四、双向复合流程

### 步骤1：建立全部制度清单并确认效力

识别现行、废止、被替代、前瞻参考和行内待确认版本。制度清单不完整时，不得宣称完成全覆盖。

### 步骤2：结构化制度源文件

- Markdown、可解析PDF和XLSX按条号拆分；
- 无条号细则按稳定章节拆分；
- 扫描件或提取异常条款进入人工原文校核；
- 印发通知正文只作为制度来源和效力记录，不重复制造项目动作。

### 步骤3：拆分原子义务

按句、款、项拆出最小义务。完整原条款始终保留在条款层，拆分不能造成上下文丢失。对纯定义、监管机关职责、解释和生效条款也保留记录，但标明“不直接生成项目任务”。

### 步骤4：制度到动作的正向分类

分类优先顺序为：

1. 已核验条款的人工映射规则；
2. 原子义务文本语义匹配；
3. 无条号行内制度的原子文本规则；
4. 章节级兜底分类；
5. 待人工归类。

不同层级必须在分类状态中明确，章节兜底不得标记为“已确认”。

### 步骤5：管理动作分类、合并和体系化整理

先按治理、战略分类、风险评估、供应商、合同、服务质量、人员、数据、AI、交付、连续性、审计报告、退出档案和问责等管理域归类，再合并同义动作。

合并时同时检查：

- 责任主体是否一致；
- 管理对象是否一致；
- 动作强度是否一致；
- 时点和频率是否一致；
- 证据是否一致；
- 适用条件是否一致。

只要上述要素存在实质差异，就应拆为不同标准动作，或者在一个动作下明确子动作，不能仅凭关键词相近合并。

### 步骤6：标准动作到执行计划的反向映射

逐项读取执行计划任务、属性、适用性、材料和制度依据，识别其对应标准动作。任务文本未匹配但制度依据能够反推时，保留“依据反推”；两条链路都有结果时，保留复合结果并抽查冲突。

### 步骤7：制度依据关系分层

依据关系统一分为：

- 外部监管直接依据；
- 行内承接直接依据；
- 采购或合同直接依据；
- 条件依据；
- 机构层面依据；
- 补充支持依据；
- 不适用或错引；
- 待核。

支持依据不能替代直接依据。机构义务不能直接改写成项目组职责，项目组只能按职责提供材料或配合。

### 步骤8：执行六类双向完整性检查

1. 正向完整性：每个原子义务至少进入一个标准动作；
2. 反向有据性：每个监管型标准动作至少有一项可追溯制度依据；
3. 执行覆盖：每个本项目适用动作有任务，或明确并入及不适用理由；
4. 合并不丢失：动作库保留全部义务ID和条款ID；
5. 适用性一致：制度、动作和任务的适用判断不冲突；
6. 五要素一致：主体、对象、动作、时点频率、证据一致。

### 步骤9：从缺口反向返回制度结构化层

发现以下问题时，不直接在执行计划末端补字，而是返回上游修正：

- 制度有义务但无动作：回到拆分和分类规则；
- 动作过宽或分类错误：回到原子义务，拆分或调整动作；
- 动作有任务但无依据：回到制度清单和条款库查漏；
- 动作有依据但无任务：判断新增、并入、机构层面或当前不适用；
- 引用在合并中丢失：停止回写，恢复义务ID并集；
- 条款和任务五要素不一致：删除、降为支持或更换依据。

这一步构成工作流的反馈闭环，不允许只在最终表格中手工掩盖上游错误。

### 步骤10：人工抽查和确认

至少抽查：全部人工覆盖规则、全部章节兜底结果、全部条件适用动作、全部机构层面动作、全部错引建议、全部合并粒度提示和全部无法解析依据。

人工确认填写最终动作、直接依据、支持依据、触发条件、不适用理由和是否允许回写。

### 步骤11：回写执行计划

只有人工确认允许回写后才修改执行计划。回写时同步核对责任主体、阶段、任务属性、适用性、时点频率、材料、关闭条件和制度依据，不只修改引用列。

### 步骤12：迭代修正规则

每轮记录自动分类命中率、待归类数量、错分、漏分、错引、过宽引用和缺任务数量。经人工确认的结论才可沉淀为稳定规则；项目特有事实不得固化为通用监管规则。

## 五、阻断规则

出现以下任一情形，不得标记完成或写回执行计划：

- 制度清单或效力状态未确认；
- 条款原文无法可靠取得；
- 存在未分类的适用义务；
- 存在合并引用丢失；
- 条款责任主体、对象或动作与任务不一致；
- 条件适用但未记录触发条件和当前判断；
- 机构层面事项被直接分派给项目团队；
- 监管、行内、采购合同和支持依据未分层；
- 拟回写内容未经人工确认。

## 六、当前项目回放

当前双向工作底稿基于9类制度或项目来源、277条或章节化要求、757项原子义务和223项执行计划任务生成。主要外部监管及行内3.0版采用逐条人工映射；无条号配套制度先按原子文本分类，无法命中时使用章节规则并标记待抽查。

首轮回放曾因低分匹配阈值和合并引用校验实现错误产生大量误报，已修复为：低分仍保留最高匹配结果；合并完整性必须从动作库实际保存的义务ID并集反查。该修正本身说明双向校验不仅检查业务内容，也检查工作流实现是否可靠。

本轮只更新复核工作流、双向工作底稿和结构化数据，不修改原执行计划。

## 七、运行方式

在本目录运行：

```bash
python3 tools/bidirectional_regulatory_workflow.py
```

生成：

- `制度依据双向复合复核工作底稿-授信系统群咨询项目.xlsx`；
- `制度依据专项复核数据/原子义务结构化.json`；
- `制度依据专项复核数据/标准管理动作库.json`；
- `制度依据专项复核数据/执行计划到标准动作映射.json`；
- `制度依据专项复核数据/双向覆盖缺口.json`。

原单向专项复核脚本和底稿继续保留，用于对照历史问题和验证双向工作流没有把既有问题静默消除。
